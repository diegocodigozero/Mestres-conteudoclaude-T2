## 2. `CLAUDE.md` (na raiz, fora de `.claude`)

````markdown
# Squad 08 — Dossiê de Concorrência (Competitive Intelligence)

Squad multi-agente que transforma **2 a 4 URLs de concorrentes** (mais,
opcionalmente, o seu próprio site) em um **dossiê comparativo estilo
consultoria**: matriz comparativa, SWOT por concorrente, gaps de mercado,
seção "você vs eles" e battle cards com ângulos de diferenciação. Reaproveita
a mesma base de **Playwright headless** do Squad 01 para "abrir" e ler os
sites dos concorrentes sozinho, e usa **WebSearch** para reputação externa
(reviews, notícias, posicionamento). **Sem nenhuma API key paga.**

## Para quem é

Founders, times de produto/marketing, consultores e agências que precisam
mapear rapidamente o terreno competitivo de um mercado, entender onde cada
concorrente é forte/fraco e sair com argumentos concretos de venda
("como ganhar de cada um") — sem pagar por ferramentas de inteligência
competitiva.

## O efeito UAU

Você cola 2-4 URLs e vê o agente **abrir os sites dos concorrentes
sozinho** (Playwright headless, sem janela aparecer), extrair proposta de
valor, ofertas, preços e provas sociais, cruzar com a reputação externa e
devolver um **relatório com cara de entregável de consultoria** em HTML
(matriz colorida, cards de SWOT, seção de oportunidades) + a versão `.md`.

## Composição do Squad

| Agente | Papel | Quando é acionado |
|--------|-------|-------------------|
| `maestro-dossie` | **Orquestrador** — onboarding, gera `<id>`, valida ambiente (via `validador`), dispara `coletor-site` EM PARALELO por URL, depois `pesquisador-externo`, depois `analista`, depois `redator-dossie`. NÃO coleta nem analisa. | `iniciar squad` |
| `validador` | **Ambiente** — checa Python + Playwright + Chromium (instala se faltar) e `output/` gravável. Roda no Passo 0. | Logo após onboarding, antes de qualquer coleta |
| `coletor-site` | **Coleta de 1 site** — recebe UMA URL, abre em Playwright headless (bloqueia mídia pesada), extrai dados estruturados. Um JSON por concorrente. | Em paralelo, uma instância por URL |
| `pesquisador-externo` | **Reputação externa** — para cada concorrente, WebSearch de reviews, reputação, notícias e posicionamento; sempre com fonte. | Após a coleta dos sites |
| `analista` | **Cruzamento** — monta matriz comparativa (atributos × concorrentes + você), SWOT por concorrente e gaps de mercado não atendidos. | Após coleta + pesquisa externa |
| `redator-dossie` | **Entregável** — battle cards (como ganhar de cada um), ângulos de diferenciação, posicionamento sugerido; gera `dossie.html` + `dossie.md`. | Após a análise |

> **Regra de ouro:** o orquestrador NUNCA executa — ele decompõe, delega,
> coordena e sintetiza. Cada subagente tem **escopo único**.

## Regras da casa (válidas para todos os agentes/skills)

1. **Orquestrador não executa.** O `maestro-dossie` só decompõe, delega,
   coordena e sintetiza.
2. **Escopo único por agente.** Coletar site, pesquisar reputação, analisar e
   redigir são agentes diferentes.
3. **Nunca inventar dados.** Campo sem fonte fica `null` (JSON) ou marcado
   `não informado` (relatório). Toda afirmação de reputação externa carrega a
   **URL da fonte**.
4. **Saída estruturada.** JSON com schema fixo na coleta/pesquisa/análise;
   markdown + HTML no entregável final.
5. **Datas nunca são inventadas.** A data vem do onboarding (campo Bônus). Se
   o usuário não informar, use o placeholder literal `<data>` no `<id>` e no
   rodapé — **não chute** uma data.
6. **Playwright sempre headless** — a janela nunca aparece; bloquear mídia
   pesada (imagens, fontes, mídia) para acelerar.
7. **Sem secrets.** Este squad não usa API key paga; não há `config/.env`.

## Convenções de saída

- `<id>` = `dossie-<segmento-slug>-<data>` (kebab-case do segmento + data

- Datas no formato `YYYY-MM-DD`.
- Numeração de concorrentes `concorrente-01`, `concorrente-02`, … na ordem

- Todos os caminhos referenciados pelos agentes são **relativos** à raiz do
  squad (`output/dossies/<id>/...`).
- `dossie.html` é o entregável final que o usuário abre no browser.

## Variáveis de ambiente
````

````markdown
# Validar Ambiente: Python + Playwright + Chromium + output/ (auto-instala)

## Quando acionar

**Sempre antes** de o `coletor-site` abrir qualquer URL. É a primeira etapa de
execução (Passo 0) após o onboarding ser confirmado.

## Procedimento detalhado

### Etapa 1 — Verificar Python
Execute `python --version` (fallback `python3 --version`).
- OK → avança.
- Falha → pare: "Python 3.10+ não encontrado. Instale antes de continuar."

### Etapa 2 — Verificar Playwright (pacote pip)
Execute `python -c "import playwright; print(playwright.__version__)"`.
- Retorna versão → OK, avança.
- `ModuleNotFoundError` → **instale automaticamente**:
  ```
  pip install playwright
  ```
  (fallback `pip3 install playwright`). Re-execute o check para confirmar.
- Se falhar (sem pip / sem internet / permissão) → pare e reporte.

### Etapa 3 — Verificar Chromium (binário do browser)
Execute:
```
python -c "from playwright.sync_api import sync_playwright; p=sync_playwright().start(); b=p.chromium.launch(headless=True); b.close(); p.stop()"
```
- Sem erro → Chromium OK.
- "Executable doesn't exist" / "browser not found" → **instale**:
  ```
  python -m playwright install chromium
  ```
  Aguarde o download (1–3 min). Não instale firefox nem webkit. Re-cheque.
- Se falhar → pare e reporte.

### Etapa 4 — Verificar pasta output/ gravável
- Garanta que `output/dossies/` existe (crie se faltar).
- Teste gravação: escreva e remova um arquivo temporário em `output/`.
- Se não conseguir gravar → pare: "Pasta output/ não gravável. Ajuste as
  permissões e rode `iniciar squad` novamente."

### Etapa 5 — Verificar acesso à internet
- O squad precisa **abrir sites** (Playwright) e **pesquisar** (WebSearch).
- Confirme conectividade (ex.: o launch do Chromium já testa o ambiente; um
  HEAD simples a um host público confirma a rede).
- Se offline → pare com aviso amigável: "Sem acesso à internet. O dossiê
  precisa abrir os sites dos concorrentes e pesquisar reputação. Conecte-se e
  rode `iniciar squad` novamente."

## Apresentação do checklist (Passo 0)

```
Passo 0 — Pré-requisitos
[OK]    Python 3.10+ encontrado
[OK]    Playwright instalado
[OK]    Chromium baixado
[OK]    Pasta output/ gravável
[OK]    Acesso à internet (sites dos concorrentes + WebSearch)
```

Use `[FALTA]` em itens pendentes; ao instalar algo, marque `[OK]` e diga o que
foi instalado (ex.: "Playwright instalado (1.x) + Chromium baixado").

## Restrições

- **Nunca pule** a validação.
- **Instala Playwright/Chromium sozinho** se faltar — não pede permissão.
- Só instala **Chromium**, nunca firefox/webkit.
- **Não navega** em site nenhum — só valida o ambiente.
- **Não há API key** neste squad — não peça nem valide secrets.
- Se a instalação falhar ou faltar internet/output, **pare** e reporte; não
  acione o coletor.

## Exemplo de uso

> Orquestrador: "Passo 0: valide o ambiente."
> Skill: roda etapas 1–5, instala Chromium que estava faltando, e responde:
> "Ambiente validado: Python 3.12 + Playwright 1.49 + Chromium baixado +
> output/ gravável + internet OK. Pronto para coletar."

## Saída

Mensagem de status (pronto / instalou X / falhou) para o orquestrador.
````

````markdown
# Ler Site de Concorrente (Playwright headless, 1 URL → 1 JSON)

## Pré-requisito

**Só roda após `validar-playwright` confirmar ambiente OK.** Se o validador
ainda não rodou, pare e acione-o primeiro.

## Quando acionar

Quando o orquestrador pedir a coleta de **um** site de concorrente (ou do
próprio site do usuário). Uma execução por URL — várias rodam em paralelo.

## Parâmetros

- `url` — a URL do site.
- `slot` — `concorrente-01`, `concorrente-02`, … ou `voce`.
- `caminho_saida` — `output/dossies/<id>/concorrentes/<slot>.json`.

## Procedimento detalhado

1. **Lance o navegador** com Playwright:
   - `headless=True` (obrigatório — a janela nunca aparece).
   - Bloqueie recursos pesados via `route`: aborte `image`, `media`, `font`
     (ganho de velocidade; o texto/preço continua acessível).
   - Use `timeout` razoável (ex.: 30s) e `wait_until="domcontentloaded"`.
2. **Abra a `url`** (home). Aguarde o conteúdo principal carregar.
3. **Visite páginas óbvias de sinal**, se existirem links fáceis no menu:
   - Preços: `/precos`, `/pricing`, `/planos`, `/plans`.
   - Produto/Sobre: `/produto`, `/sobre`, `/about`.
   Não faça crawl profundo — 1 a 3 páginas-chave bastam.
4. **Extraia SOMENTE o que está visível** (texto real da página):
   - `nome_marca`, `headline` (hero), `subheadline`.
   - `ofertas` (produtos/serviços listados).
   - `precos` (lista de `{plano, valor, obs}` exatamente como no site). Sem
     preço visível → `[]`.
   - `ctas` (textos dos botões principais).
   - `provas_sociais` (depoimentos, números, logos, prêmios/selos escritos).
   - `canais_contato` (`email`, `telefone`, `redes[]`, `formulario` bool).
   - `idioma` do site.
5. **Grave** o JSON no schema fixo (abaixo) em `caminho_saida`.
6. **Feche** o browser (libere recursos). Reporte 1 linha de status.

## Schema do JSON

```json
{
  "slot": "concorrente-01",
  "url": "https://concorrente-a.com",
  "coletado_em": "<data>",
  "nome_marca": "Concorrente A",
  "headline": "...",
  "subheadline": "...",
  "ofertas": ["..."],
  "precos": [{ "plano": "...", "valor": "...", "obs": "..." }],
  "ctas": ["..."],
  "provas_sociais": ["..."],
  "canais_contato": { "email": null, "telefone": null, "redes": [], "formulario": false },
  "idioma": "pt-BR",
  "observacoes": "..."
}
```

## Restrições

- **NÃO invente** dados. Campo sem evidência no site → `null` (ou `[]`).
- Preço só entra se **visível** no site. Não estime preço.
- **Sempre headless** — janela nunca aparece.
- **Bloqueie** imagens/fontes/mídia.
- Não pesquise reviews/notícias (isso é do `pesquisar-reputacao-externa`).
- Não compare nem pontue (isso é do `montar-matriz-swot`).
- Se o site bloquear/timeout, salve o que obteve, marque o resto `null` e
  registre em `observacoes`. Não preencha "no chute".
- Use **caminho relativo**.

## Exemplo de uso

> coletor-site recebe `url=https://concorrente-a.com`, `slot=concorrente-01`.
> Abre headless, lê hero + página /planos, extrai headline, 2 planos de preço,
> 1 CTA e "+2.000 clínicas" como prova social; telefone não aparece → `null`.
> Grava `output/dossies/<id>/concorrentes/concorrente-01.json` e responde:
> "Coleta de Concorrente A concluída (10 campos, 2 null)."

## Saída

Um JSON por URL em `caminho_saida` + 1 linha de status no chat.
````

````markdown
# Pesquisar Reputação Externa (WebSearch, 1 JSON por concorrente)

## Quando acionar

Após o `coletor-site` gravar os JSONs de coleta, para enriquecer cada
concorrente com sinais que **não estão no próprio site** (o que o mercado
diz sobre ele).

## Parâmetros

- Lista de concorrentes: `slot` + `nome_marca` + `url`.
- `segmento` e `geografia` (do briefing) para refinar idioma e termos.
- Diretório de saída `output/dossies/<id>/reputacao/`.

## Procedimento detalhado

Para **cada concorrente**:

1. **Monte as consultas WebSearch** (ajuste idioma à `geografia`):
   - `"<nome_marca>" reviews` / `"<nome_marca>" avaliações`
   - `"<nome_marca>" reclame aqui` (quando geografia = Brasil)
   - `"<nome_marca>" <segmento> opinião / é bom`
   - `"<nome_marca>" alternativa / vs`
   - `"<nome_marca>" notícia <ano-da-data-de-referência>`
   - `"<nome_marca>" clientes / posicionamento`
2. **Rode o WebSearch** e selecione os resultados mais relevantes e
   confiáveis (priorize fontes reconhecíveis: portais, sites de review,
   imprensa). Descarte spam e conteúdo claramente promocional disfarçado.
3. Para cada achado, registre:
   - `tipo`: `review` | `reputacao` | `noticia` | `posicionamento`
   - `resumo`: 1 frase objetiva (não copie parágrafos)
   - `sentimento`: `positivo` | `neutro` | `negativo` | `misto`
   - `fonte`: a **URL** do resultado
4. **Categorias** sem achado confiável → lista **vazia** (não invente).
5. **Grave** `output/dossies/<id>/reputacao/<slot>-reputacao.json` no schema.

## Schema do JSON (por concorrente)

```json
{
  "slot": "concorrente-01",
  "nome_marca": "Concorrente A",
  "pesquisado_em": "<data>",
  "reviews":        [ { "tipo": "review",        "resumo": "...", "sentimento": "misto",    "fonte": "https://..." } ],
  "reputacao":      [ { "tipo": "reputacao",      "resumo": "...", "sentimento": "neutro",   "fonte": "https://..." } ],
  "noticias":       [ { "tipo": "noticia",        "resumo": "...", "sentimento": "positivo", "fonte": "https://..." } ],
  "posicionamento": [ { "tipo": "posicionamento", "resumo": "...", "sentimento": "neutro",   "fonte": "https://..." } ],
  "observacoes": "..."
}
```

## Restrições

- **Todo** achado carrega `fonte` (URL real). Sem fonte → não entra.
- **NÃO invente** reviews, números ou notícias.
- **NÃO copie** textos longos — resuma.
- **NÃO abra** o site do concorrente (isso é do `ler-site-concorrente`).
- **NÃO compare nem analise** entre concorrentes (isso é do `montar-matriz-swot`).
- **NÃO invente datas** — use o ano da data de referência do briefing como
  filtro; se não houver data, omita o filtro de ano.
- Use **caminhos relativos**.

## Exemplo de uso

> pesquisador-externo busca "Concorrente A reviews", encontra uma página de
> reviews com nota 3,5 citando lentidão; registra `{tipo: review, resumo:
> "Elogios ao preço, críticas à lentidão", sentimento: misto, fonte: <url>}`.
> Sem notícias confiáveis recentes → `noticias: []`. Grava o JSON.

## Saída

Um JSON por concorrente em `output/dossies/<id>/reputacao/` + status no chat.
````

````markdown
# Montar Matriz Comparativa + SWOT + Gaps de Mercado

## Quando acionar

Após existirem todos os JSONs de coleta (`concorrentes/*.json`) e reputação
(`reputacao/*.json`). Transforma dados crus em análise estruturada.

## Parâmetros

- Diretório de coleta `output/dossies/<id>/concorrentes/*.json`.
- Diretório de reputação `output/dossies/<id>/reputacao/*.json`.
- `objetivo` (posicionamento / preço / ataque) — orienta a ênfase e a
  relevância dos gaps.
- Caminho de saída `output/dossies/<id>/analise.json`.

## Procedimento detalhado

1. **Carregue** todos os JSONs de coleta e reputação, casando por `slot`.
2. **Matriz comparativa** — linhas = atributos, colunas = cada concorrente +
   `voce` (se houver `voce.json`). Atributos padrão:
   - `proposta_de_valor` (de `headline`/`subheadline`)
   - `ofertas_principais`
   - `faixa_preco` (derivada de `precos`; se vazio → "não informado")
   - `modelo_preco` (assinatura mensal / único / freemium / "não informado")
   - `cta_principal`
   - `prova_social_forte` (a prova social mais relevante)
   - `reputacao_sentimento` (sentimento dominante dos achados externos)
   - `canais_contato`
   - `geografia_idioma`
   - Célula sem evidência → `"não informado"`.
3. **SWOT por concorrente** (`forcas`, `fraquezas`, `oportunidades`,
   `ameacas`):
   - Forças/fraquezas: do site (ofertas, preço, CTA, provas) + reputação.
   - Oportunidades/ameaças: do mercado (gaps que o concorrente deixa, notícias).
   - **Todo** item vindo de reputação externa anexa `— fonte: <url>`.
4. **Gaps de mercado** — necessidades/ângulos que **nenhum** concorrente
   atende bem. Para cada gap: `gap`, `evidencia` (o que nos dados sustenta o
   gap) e `relevancia_para_objetivo` (`alta`/`média`/`baixa`) à luz do
   `objetivo` do dossiê.
5. **Grave** `analise.json` no schema fixo.

## Schema do JSON

```json
{
  "id": "dossie-<segmento-slug>-<data>",
  "analisado_em": "<data>",
  "objetivo": "posicionamento",
  "atributos": ["proposta_de_valor","ofertas_principais","faixa_preco","modelo_preco","cta_principal","prova_social_forte","reputacao_sentimento","canais_contato","geografia_idioma"],
  "matriz": {
    "voce":          { "...": "não informado" },
    "concorrente-01":{ "proposta_de_valor": "...", "faixa_preco": "R$ 99–199/mês" },
    "concorrente-02":{ "...": "..." }
  },
  "swot": {
    "concorrente-01": {
      "forcas": ["..."],
      "fraquezas": ["App lento — fonte: https://..."],
      "oportunidades": ["..."],
      "ameacas": ["Rodada seed — fonte: https://..."]
    }
  },
  "gaps_mercado": [
    { "gap": "...", "evidencia": "...", "relevancia_para_objetivo": "alta" }
  ]
}
```

## Restrições

- **NUNCA invente.** Célula sem evidência → `"não informado"`.
- **Toda** afirmação de SWOT vinda de reputação carrega a **fonte**.
- **NÃO escreva** battle cards/posicionamento/HTML (isso é do redator).
- **NÃO recolete** sites nem refaça WebSearch — só os JSONs existentes.
- Coluna `voce` só preenchida se houver `voce.json`.
- Use **caminhos relativos**.

## Exemplo de uso

> analista carrega 3 coletas + 3 reputações. Monta matriz 9×4 (3 concorrentes
> + você), gera SWOT dos 3 e identifica 2 gaps ("ninguém oferece onboarding
> assistido"; "faixa abaixo de R$ 80 descoberta"). Grava `analise.json`.

## Saída

`output/dossies/<id>/analise.json` + status no chat (dimensões da matriz, nº de
SWOTs e de gaps).
````

## 6. Secrets — por que NÃO há `config/.env`

Este squad **não usa nenhuma API key paga**. Ele usa apenas:

- **Playwright** (local, gratuito) para abrir os sites dos concorrentes.
- **WebSearch** (ferramenta nativa do Claude Code) para a reputação externa.

Portanto **não existe `config/.env` nem `config/.env.example`**. O `.gitignore`
ainda lista `config/.env` por segurança (caso alguém adicione integração no
futuro), mas nenhuma chave é necessária para rodar o squad.

> Regra da casa: secrets, quando existirem, ficam só em `config/.env`
> (carregado via python-dotenv), nunca hardcoded, nunca versionados. Aqui não
