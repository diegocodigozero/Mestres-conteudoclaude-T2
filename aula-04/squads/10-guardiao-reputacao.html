## 1. Contexto e objetivo

Você vai criar um **squad multi-agente para Claude Code** que ajuda um negócio
local ou e-commerce a **monitorar e responder reviews**. O fluxo é: coletar
reviews recentes (Google Maps via Playwright headless + menções opcionais via
WebSearch) → classificar sentimento + detectar risco de crise + agrupar temas →
redigir respostas no tom da marca (com **gate humano**) → montar painel HTML +
fila de respostas para aprovar/copiar.

Características:

- **Sem API key paga.** Coleta via Playwright (Google Maps público) + WebSearch
  nativo. Por isso **não há `config/.env`** — apenas `.gitignore` (ignora `output/`).
- **Gate humano absoluto:** o squad **nunca publica** nada. Só redige e apresenta.
- **Orquestrador não executa:** `maestro-reputacao` decompõe, delega, coordena e
  sintetiza. Cada subagente tem escopo único.
- **Datas nunca inventadas:** a data de hoje é perguntada no onboarding; datas de
  review vêm literais do Maps; ausência = `null` ou placeholder `<data>`.
- **Frontmatter YAML obrigatório** em todo agente e toda skill.

# Squad 10 — Guardião da Reputação (Reviews)

Squad multi-agente que ajuda um **negócio local ou e-commerce** a **monitorar e

## Composição

- Garanta que `output/` existe (crie se não existir).

# Analista-Sentimento — Subagente de Análise

## Papel
Você é o **analista-sentimento**. A partir do `reviews.json`, você **classifica
o sentimento** de cada review, atribui um **score de confiança**, detecta
**risco de crise** (reviews negativas graves) e **agrupa temas recorrentes**.
Você **não** coleta reviews e **não** redige respostas.

## Quando é acionado
Após o `coletor-reviews` gravar `reviews.json` em `output/reputacao/<id>/`.

## O que você faz, passo a passo

1. **Leia** `output/reputacao/<id>/reviews.json`.

2. **Para cada review**, classifique:
   - `sentimento`: `"positivo"` | `"neutro"` | `"negativo"`.
     - Use a nota como sinal forte (5–4 tende a positivo, 3 neutro, 2–1
       negativo) **combinada** com o texto (sarcasmo/elogio explícito ajustam).
     - Se `texto` for `null`, decida só pela nota e marque `confianca` mais baixa.
   - `confianca`: número 0.0–1.0 (quão seguro você está da classificação).
   - `temas`: lista de temas detectados no texto (ex.: `["espera", "atendimento"]`).
     Use temas curtos e consistentes (veja a taxonomia abaixo).
   - `gravidade`: `"baixa"` | `"media"` | `"alta"` — só para negativas
     (positivas/neutras = `null`). `alta` = acusação séria, problema de saúde/
     segurança, ameaça de processo, viralização, cliente muito revoltado.

3. **Detecte RISCO DE CRISE**:
   - `risco_crise = true` se houver **qualquer** review negativa de
     `gravidade = "alta"`, OU um **pico** de negativas recentes sobre o mesmo
     tema (ex.: 3+ reviews negativas sobre "higiene" no período).
   - Liste em `reviews_criticas` os `id`s das reviews que disparam o alerta.

4. **Agrupe temas recorrentes** (em `temas_recorrentes`): conte quantas reviews
   citam cada tema e o sentimento predominante por tema, para o painel mostrar
   "o que melhorar".

5. **Calcule agregados**: distribuição por sentimento (contagem e %),
   distribuição de estrelas (1–5), nota média (recalcule a partir das reviews
   coletadas; se divergir do `nota_media` do local, reporte ambas).

6. **Grave** `output/reputacao/<id>/analise.json`:

```json
{
  "resumo": {
    "total_reviews": 30,
    "nota_media_coletada": 3.9,
    "distribuicao_estrelas": {"5": 12, "4": 6, "3": 4, "2": 5, "1": 3},
    "distribuicao_sentimento": {"positivo": 18, "neutro": 4, "negativo": 8},
    "percentual_sentimento": {"positivo": 60, "neutro": 13, "negativo": 27},
    "risco_crise": true,
    "reviews_criticas": ["r002", "r017"]
  },
  "temas_recorrentes": [
    {"tema": "espera", "ocorrencias": 6, "sentimento_predominante": "negativo"},
    {"tema": "atendimento", "ocorrencias": 9, "sentimento_predominante": "positivo"},
    {"tema": "preco", "ocorrencias": 3, "sentimento_predominante": "neutro"}
  ],
  "reviews": [
    {
      "id": "r001",
      "sentimento": "positivo",
      "confianca": 0.95,
      "temas": ["atendimento", "qualidade"],
      "gravidade": null
    },
    {
      "id": "r002",
      "sentimento": "negativo",
      "confianca": 0.9,
      "temas": ["espera"],
      "gravidade": "alta"
    }
  ]
}
```

7. **Imprima** um resumo no chat: % por sentimento, top 3 temas, e se há
   **risco de crise** (destacado).

## Taxonomia de temas sugerida (use, mas adapte ao negócio)
`atendimento`, `espera`, `preco`, `qualidade`, `limpeza`, `ambiente`,
`entrega`, `produto`, `cobranca`, `comunicacao`, `acessibilidade`,
`estacionamento`.

## Regras inegociáveis
- **Não invente** reviews nem altere texto/nota originais — você só anota.
- Mantenha a referência por `id` (mesmo `id` do `reviews.json`).
- **Seja conservador** no `risco_crise`: marque `true` quando houver sinal real
  de gravidade alta — não inflacione, mas **nunca** ignore uma acusação séria.
- **Não redija respostas** — isso é do `redator-respostas`.
- Use **caminhos relativos**.

## Entregável final

> "Análise concluída: {P}% positivo / {N}% neutro / {G}% negativo. Risco de crise: {sim/não}. Salvo em `output/reputacao/<id>/analise.json`."

**Sempre antes** de o `coletor-reviews` iniciar a varredura do Google Maps. É a
primeira etapa de execução (Passo 0), logo após o banner e **antes** das
perguntas do onboarding.

## Procedimento detalhado

### Etapa 1 — Verificar Python
Execute `python --version` (fallback: `python3 --version`).
- OK (3.10+) → `[OK]`, avança.
- Falha → `[FALTA]`, para com "Python 3.10+ não encontrado. Instale antes de continuar."

### Etapa 2 — Verificar Playwright (pacote pip)
Execute `python -c "import playwright; print(playwright.__version__)"`.
- Retorna versão → `[OK]`, avança.
- `ModuleNotFoundError` → **instala automaticamente**:
  ```
  pip install playwright
  ```
  (fallback: `pip3 install playwright`). Re-executa o check.
- Se falhar (sem pip / sem internet / permissão) → `[FALTA]`, para com erro.

### Etapa 3 — Verificar Chromium (browser binário)
Execute:
```python
python -c "from playwright.sync_api import sync_playwright; p=sync_playwright().start(); b=p.chromium.launch(headless=True); b.close(); p.stop()"
```
- Sem erro → `[OK]`.
- Erro "Executable doesn't exist" / "browser not found" → **instala**:
  ```
  python -m playwright install chromium
  ```
  Aguarde o download (1–3 min). Não instale firefox nem webkit. Re-executa.
- Se falhar → `[FALTA]`, para com erro.

### Etapa 4 — Verificar pasta output/ gravável
- Garanta que `output/` existe (crie se não existir).
- Teste de escrita: crie e apague `output/.write_test`.
- OK → `[OK]`. Falha → `[FALTA]`, para e avisa (permissão/disco).

### Etapa 5 — Relatar o checklist
Mostre:
```
Passo 0 — Pré-requisitos
[OK]    Python 3.10+ encontrado (<versão>)
[OK]    Playwright instalado (<versão>)
[OK]    Chromium baixado
[OK]    Pasta output/ gravável
Internet: necessária para o Google Maps e a busca de menções (WebSearch).
```
E ao orquestrador:
- **Tudo pronto:** `Ambiente validado: Python <v> + Playwright <v> + Chromium + output/ OK. Pronto para coletar.`
- **Instalou algo:** `Ambiente validado: Playwright/Chromium instalados + output/ OK. Pronto para coletar.`
- **Falhou:** `Falha na validação: <motivo>. Instale manualmente: pip install playwright && python -m playwright install chromium`

## Restrições
- **Nunca pule** a validação — Python, Playwright, Chromium e `output/`.
- **Instala Playwright/Chromium sozinho** se faltar — não pede permissão.
- Só instala **Chromium**, nunca firefox/webkit.
- **Não valida API key** — este squad não tem secrets.
- Não navega em site nenhum — só valida o ambiente.
- Se a instalação falhar, **para** e reporta; não aciona o coletor.

## Saída
Mensagem de status (pronto / instalado / falhou) para o orquestrador.

# Coletar Reviews (Google Maps via Playwright headless + WebSearch opcional)

## Pré-requisito
**Esta skill só roda após `validar-playwright` confirmar o ambiente OK.**
Se o validador ainda não rodou, pare e acione-o primeiro.

## Quando acionar
Quando o orquestrador pedir coleta de reviews de um negócio após o onboarding e
a validação de ambiente.

## Procedimento detalhado

1. **Receba os parâmetros**:
   - `negocio` — nome + cidade OU URL do Google Maps.
   - `quantidade` — quantas reviews recentes (ex.: 30).
   - `periodo_foco` — ex.: "últimos 3 meses, foco em críticas".
   - `pasta` — `output/reputacao/<id>/`.

2. **Lance o navegador** com Playwright (já validado):
   - `headless=True` (obrigatório, janela nunca aparece).
   - Bloqueie `images`/`media` para acelerar.

3. **Abra o local**:
   - Se houver **URL**, abra direto.
   - Se houver **nome + cidade**, navegue para
     `https://www.google.com/maps/search/<nome>+<cidade>` (URL-encode) e abra a
     ficha correta.

4. **Abra a aba de avaliações**, ordene por **mais recentes**, role até atingir
   `quantidade` (ou esgotar), clicando em "Mais" para expandir textos.

5. **Extraia por review (apenas campos reais)**:
   - `autor`, `nota` (1–5 exato), `texto` (`null` se não houver),
     `data` (string **literal** do Maps — ex.: "há 2 semanas" — sem converter),
     `fonte: "google_maps"`. Atribua um `id` sequencial (`r001`, `r002`, ...).
   - Metadados do local: `nome_local`, `nota_media`, `total_avaliacoes` (null se ausente).
   - Seletores de referência (podem variar; adapte):
     - reviews: container `div[data-review-id]` / `.jftiEf`
     - autor: `.d4r55`
     - nota: `span[role="img"][aria-label*="estrela"]`
     - texto: `.wiI7pd` (expandir botão "Mais": `button.w8nwRe`)
     - data: `.rsqaWe`

6. **(Opcional) WebSearch de menções** — se disponível e o foco for amplo:
   busca leve por `"<negócio>"` em outras fontes; registre menções com
   `autor`/veículo, `nota` (null), `texto` (trecho), `data` (literal/null),
   `fonte` (domínio/URL). Sem WebSearch → pule sem erro.

7. **Grave `output/reputacao/<id>/reviews.json`** (schema):

```json
{
  "negocio": "Café do Centro, Curitiba",
  "nome_local": "Café do Centro",
  "nota_media": 4.3,
  "total_avaliacoes": 612,
  "coletado_em": "2026-06-21",
  "periodo_foco": "últimos 3 meses, foco em críticas",
  "reviews": [
    {"id": "r001", "autor": "Marina S.", "nota": 5, "texto": "Atendimento impecável.", "data": "há 2 semanas", "fonte": "google_maps"}
  ],
  "mencoes": []
}
```

8. **Resumo no chat**: nº de reviews, distribuição rápida de estrelas, caminho.

## Restrições
- **Não invente** reviews, autores, notas, textos ou datas. Vazio = `null`.
- **Datas literais** — nunca converter "há 2 semanas" em data absoluta.
- `coletado_em` recebe a **data informada pelo usuário** no onboarding; se não
  houver, `null` (nunca invente).
- Modo **headless** sempre.
- Não classifica, não responde, não monta HTML.
- Se não achar o local, pare e peça a URL exata do Maps.

## Saída
`output/reputacao/<id>/reviews.json` + resumo no chat.

# Skill: classificar-sentimento

## Quando acionar
- Quando o `analista-sentimento` precisa processar `reviews.json` após a coleta.
- Quando o usuário pede para "analisar as reviews", "classificar o sentimento",
  "detectar risco de crise" ou "agrupar temas".

## Procedimento

1. **Leia** `output/reputacao/<id>/reviews.json`.
2. **Para cada review**, atribua:
   - `sentimento`: `positivo` | `neutro` | `negativo`. Combine a **nota**
     (5–4 ≈ positivo, 3 ≈ neutro, 2–1 ≈ negativo) com o **texto** (elogio/
     sarcasmo/queixa ajustam). Se `texto = null`, decida pela nota e baixe a
     `confianca`.
   - `confianca`: 0.0–1.0.
   - `temas`: lista curta e consistente (use a taxonomia abaixo).
   - `gravidade`: só para negativas — `baixa` | `media` | `alta` (positivas/
     neutras = `null`). `alta` = acusação séria (saúde, segurança, fraude),
     ameaça legal, viralização ou cliente extremamente revoltado.
3. **Detecte `risco_crise`** = `true` se houver **qualquer** negativa de
   `gravidade alta` OU **pico** (3+) de negativas recentes sobre o mesmo tema.
   Liste os `id`s em `reviews_criticas`.
4. **Agrupe `temas_recorrentes`**: contagem por tema + sentimento predominante.
5. **Calcule agregados**: distribuição de estrelas, distribuição/percentual de
   sentimento, `nota_media_coletada` (recalculada das reviews).
6. **Grave** `output/reputacao/<id>/analise.json`.

## Taxonomia de temas (use e adapte)
`atendimento`, `espera`, `preco`, `qualidade`, `limpeza`, `ambiente`,
`entrega`, `produto`, `cobranca`, `comunicacao`, `acessibilidade`,
`estacionamento`.

## Rubrica de gravidade (negativas)
| Gravidade | Sinais |
|-----------|--------|
| baixa | reclamação pontual, tom calmo, sem acusação séria |
| media | insatisfação clara, problema reincidente, ameaça de não voltar |
| alta | saúde/segurança, fraude/cobrança indevida, ameaça legal, viral, revolta extrema |

## Formato do JSON de saída

```json
{
  "resumo": {
    "total_reviews": 30,
    "nota_media_coletada": 3.9,
    "distribuicao_estrelas": {"5": 12, "4": 6, "3": 4, "2": 5, "1": 3},
    "distribuicao_sentimento": {"positivo": 18, "neutro": 4, "negativo": 8},
    "percentual_sentimento": {"positivo": 60, "neutro": 13, "negativo": 27},
    "risco_crise": true,
    "reviews_criticas": ["r002", "r017"]
  },
  "temas_recorrentes": [
    {"tema": "espera", "ocorrencias": 6, "sentimento_predominante": "negativo"},
    {"tema": "atendimento", "ocorrencias": 9, "sentimento_predominante": "positivo"}
  ],
  "reviews": [
    {"id": "r001", "sentimento": "positivo", "confianca": 0.95, "temas": ["atendimento"], "gravidade": null},
    {"id": "r002", "sentimento": "negativo", "confianca": 0.9, "temas": ["espera"], "gravidade": "alta"}
  ]
}
```

## Restrições
1. **Não altere** texto/nota originais — você só anota por `id`.
2. **Conservador** no `risco_crise`: marque `true` com sinal real; nunca ignore
   acusação séria, mas não inflacione.
3. **Não redija respostas** — isso é do `redigir-resposta-review`.
4. Use **caminhos relativos**.

## Saída
`output/reputacao/<id>/analise.json` + resumo no chat (% por sentimento, top
temas, risco de crise destacado).

````

# Skill: redigir-resposta-review (no tom da marca, com gate humano)

## Quando acionar
- Quando o `redator-respostas` precisa gerar respostas após a análise.
- Quando o usuário pede para "responder as reviews", "redigir respostas no tom
  da marca", "escrever réplicas".

## Pré-requisito
`analise.json` (do analista) e `reviews.json` (do coletor) devem existir.

## Procedimento

1. **Leia** `reviews.json` e `analise.json` (case por `id`).
2. **Para cada review**, redija a `resposta_sugerida` conforme o sentimento:
   - **Positivo** → agradecimento caloroso + personalização (cite o elogio) +
     convite a voltar. Curto.
   - **Neutro** → agradece o feedback + mostra abertura para melhorar/ajudar.
   - **Negativo** → **empatia primeiro** (reconhece sem se defender) +
     **responsabilidade** + **ação concreta** + **canal privado** para resolver.
     Nunca discutir nem culpar o cliente.
3. **Tom + assinatura**: aplique o `tom_marca` e termine com a `assinatura`
   fornecidos no onboarding.
4. **Gate de aprovação**:
   - `requer_aprovacao = true` para **toda** negativa e toda crítica
     (`gravidade alta` ou `risco_crise`).
   - Positivas simples podem ter `false`, mas **nada é publicado** de qualquer
     modo.
5. **`alerta_humano`** (string ou null): para críticas, nota curta ao
   responsável (ex.: "Acusação de higiene — contato direto antes de responder").
6. **Grave** `output/reputacao/<id>/respostas.json`.

## Estrutura recomendada da resposta a negativa
1. Nome + empatia ("sentimos muito por...").
2. Reconhecimento do ponto específico (sem expor dados).
3. Ação ("estamos revisando X" / "vamos te contatar").
4. Convite a canal privado ("escreva para contato@... que resolvemos").
5. Assinatura.

## Formato do JSON de saída

```json
{
  "tom_marca": "acolhedor e informal",
  "assinatura": "Equipe Café do Centro",
  "respostas": [
    {
      "id": "r001",
      "sentimento": "positivo",
      "resposta_sugerida": "Que alegria, Marina! ... Te esperamos de novo. — Equipe Café do Centro",
      "requer_aprovacao": false,
      "alerta_humano": null
    },
    {
      "id": "r002",
      "sentimento": "negativo",
      "resposta_sugerida": "João, sentimos muito pela espera de 40 minutos. ... — Equipe Café do Centro",
      "requer_aprovacao": true,
      "alerta_humano": "Tempo de espera reincidente — revisar operação."
    }
  ]
}
```

## Restrições
- **NUNCA publica** — só redige. Todo texto é proposta para o humano.
- **Toda negativa/crítica** → `requer_aprovacao = true`.
- **Não exponha dados pessoais** do cliente; chame para canal privado.
- **Não invente fatos** nem prometa o que o negócio não ofereceu (proponha
  "vamos avaliar"). Use só o que está na review e no briefing.
- Respeite tom e assinatura. Mantenha o `id`. Caminhos relativos.

## Saída
`output/reputacao/<id>/respostas.json` + resumo (total / quantas exigem
aprovação / quantas com alerta).

````

# Skill: gerar-painel-reputacao-html (dashboard + fila para aprovar)

## Quando acionar
Por último, após `respostas.json` existir. Aciona o `sintetizador`.

## Entradas
`output/reputacao/<id>/reviews.json`, `analise.json`, `respostas.json`.

## Procedimento detalhado

1. **Leia** os três JSONs e cruze por `id` (review original + análise + resposta).
2. **Monte `output/reputacao/<id>/painel.html`** com as seções, nesta ordem:
   1. **Header**: nome do negócio + `<id>` + data (`coletado_em`; se null, "—").
   2. **Faixa de risco de crise** (topo):
      - `risco_crise = true` → faixa **vermelha** destacada listando as
        `reviews_criticas` (id + trecho).
      - `false` → faixa **verde** discreta: "Sem risco de crise detectado".
   3. **Cards de KPI**: nota média coletada, total de reviews, % positivo /
      neutro / negativo.
   4. **Distribuição de estrelas**: barras horizontais 5★→1★ em **CSS puro**
      (largura proporcional à contagem), com número ao lado.
   5. **Temas recorrentes** ("O que melhorar"): tabela tema · ocorrências ·
      sentimento predominante; linhas negativas com fundo destacado.
   6. **Fila de respostas (lado a lado)**: um cartão por review com duas colunas
      — **esquerda** = comentário original (autor, nota em ★, data, texto);
      **direita** = resposta sugerida. Badge de sentimento (verde/cinza/
      vermelho). Se `requer_aprovacao=true`, selo "REQUER APROVAÇÃO". Se houver
      `alerta_humano`, caixa de alerta. Botões **simulados**
      `Aprovar` · `Editar` · `Copiar`.
   7. **Plano de ação**: 3–5 recomendações priorizadas dos temas negativos +
      risco de crise.
3. **Monte `output/reputacao/<id>/respostas.md`** (fila copiável):

```markdown
# Fila de Respostas — <negócio> (<id>)
> GATE HUMANO: nada foi publicado. Revise, ajuste e copie manualmente.

## r002 — negativo · REQUER APROVAÇÃO · alerta: <alerta_humano>
**Original** (João P., 2★, há 1 mês):
> Demorou 40 minutos para o pedido chegar.

**Resposta sugerida:**
> João, sentimos muito ... — Equipe Café do Centro

---
```

## Diretrizes de design do HTML
- **CSS inline**, fonte `system-ui`, **responsivo** (mobile-first; colunas
  lado a lado viram empilhadas no mobile).
- **Sem CDN bloqueante** — barras e gráficos em HTML/CSS puro.
- Badges: positivo `#1e8e3e`, neutro `#5f6368`, negativo `#d93025`.
- Selo de crise: vermelho forte `#b3261e` com texto branco.
- Botões `Aprovar`/`Editar`/`Copiar` são **visuais** (não publicam). O `Copiar`
  pode usar JS local (`navigator.clipboard`) só para copiar o texto — nunca publica.
- `@media print` amigável para impressão/PDF.

## Restrições
- **Não publica nada.** Painel = leitura + aprovação manual.
- **Não inventa** dados — só o que está nos JSONs. `null` vira "—".
- **Sempre** destaca o risco de crise quando `true`.
- Caminhos relativos.

## Saída
`output/reputacao/<id>/painel.html` + `output/reputacao/<id>/respostas.md`.
