## 1. Contexto e regras de arquitetura

Squad multi-agente que responde comentários do Instagram do perfil simulado
**@diegocaldas.x**. Fluxo: **coletor** puxa comentários não respondidos via
actions nomeadas do Composio → **classificador** separa em 5 categorias
(pergunta, elogio, crítica, spam, compra) com score de confiança → **redator**
redige a resposta no tom da marca → **sinalizador** esconde spam, libera
auto-respostas e marca revisão humana, gerando um dashboard HTML. A publicação só
ocorre após comando humano explícito ("publicar").

Regras obrigatórias:
- `CLAUDE.md` na **raiz** (fora de `.claude/`). Nunca criar CLAUDE.md dentro de `.claude/`.
- TODO agente e TODA skill começam com **frontmatter YAML** (`name` + `description`).
- Orquestrador NUNCA executa: decompõe, delega, coordena, sintetiza. Cada
  subagente tem escopo único; skills são procedurais (COMO), agentes executam.
- Saída estruturada (JSON/markdown). Nunca inventar dados: campo sem fonte fica null.
- Secrets só em `config/.env` (via `python-dotenv`). `config/.env.example` tem
  placeholder VAZIO. `.gitignore` cobre `config/.env` e `output/`.
- Integração com Instagram é **exclusivamente via Composio**, usando **actions
  nomeadas** (`INSTAGRAM_GET_MEDIA`, `INSTAGRAM_GET_COMMENTS`,
  `INSTAGRAM_GET_COMMENT_REPLIES`, `INSTAGRAM_REPLY_TO_COMMENT`) — nunca paths
  REST crus do Graph.
- O campo `respondido` é **sintético** (a API não retorna isso): derivado de
  `output/respondidos.json` ou de replies do próprio business.

## Integração com o Instagram (Composio — actions nomeadas)

Este squad usa **exclusivamente o Composio**, chamando **actions/tools nomeadas**
(via SDK `composio-core` ou MCP) — nunca paths REST crus do Graph. O Composio faz
o proxy autenticado para o Graph; o aluno chama sempre a action pelo nome:

| Objetivo | Action nomeada |
|----------|----------------|
| Listar mídias recentes | `INSTAGRAM_GET_MEDIA` |
| Listar comentários de uma mídia | `INSTAGRAM_GET_COMMENTS` |
| Listar replies de um comentário | `INSTAGRAM_GET_COMMENT_REPLIES` |
| Responder um comentário | `INSTAGRAM_REPLY_TO_COMMENT` |

> Os slugs exatos podem variar conforme o toolkit instalado. Confirme com
> `client.tools.get(toolkits=["INSTAGRAM"])` (SDK) ou listando as tools do
> toolkit Instagram no MCP, e chame pelo nome.

### Campo "respondido" é SINTÉTICO

A Graph API **não** retorna `responded`/`is_replied`. O squad **calcula** esse
estado localmente: mantém `output/respondidos.json` com os ids já respondidos
(atualizado pela skill `publicar-resposta-ig`) e filtra contra ele; como
alternativa, inspeciona replies via `INSTAGRAM_GET_COMMENT_REPLIES` e considera
respondido se houver reply do próprio business. `respondido` nunca vem da API.

## Secrets / Variáveis de Ambiente

O usuário já tem conta no Composio e o Instagram já está conectado no painel. O
Composio resolve internamente o business ID e o token de acesso. Portanto:

- Secrets vivem **apenas** em `config/.env` (carregado via `python-dotenv`).
- `config/.env.example` contém o placeholder VAZIO:
  \`\`\`
  COMPOSIO_API_KEY=
  \`\`\`
- `config/.env` (com a chave real) e `output/` estão no `.gitignore`. Nunca
  versione a chave nem hardcode no código.
- Não incluir `INSTAGRAM_BUSINESS_ID` nem `INSTAGRAM_ACCESS_TOKEN` em nenhum
  arquivo do squad.

## Fluxo de Execução

\`\`\`
Passo 0 (validar) → coletor → classificador → redator → sinalizador → aprovação humana → publicar
\`\`\`

# Coletor — Subagente de Coleta de Comentários Instagram

## Papel

Você é o **coletor**, primeiro subagente do Squad Instagram. Sua responsabilidade
é puxar comentários **não respondidos** dos posts recentes do @diegocaldas.x via
Composio (que já tem a conta Instagram conectada internamente) e entregá-los em
JSON estruturado para o próximo agente (classificador).

## Quando é acionado

- Após o Passo 0 (`validar-composio-instagram`) e a confirmação dos parâmetros.
- Como primeiro passo do fluxo: `coletor → classificador → redator → sinalizador`.
- Sempre que o usuário pedir "coletar comentários" ou "atualizar feed".

## Passo a passo

1. Garantir que o Passo 0 já validou `COMPOSIO_API_KEY` (em `config/.env`) e a
   conexão Instagram. O Composio resolve o business ID e o token internamente —
   não há business ID nem access token neste squad.
2. **Modo DEMO / dry-run**: se o usuário escolheu modo demo (sem conta real),
   NÃO chamar o Composio. Carregar o fixture `output/comentarios-exemplo.json`,
   tratá-lo como a coleta bruta e seguir para os passos 4–6.
3. **Modo real**: invocar a skill `composio-instagram-comentarios` para:
   a. Listar as últimas N mídias (default 20) via action nomeada
      `INSTAGRAM_GET_MEDIA` (limit=N).
   b. Para cada mídia, listar comentários via action `INSTAGRAM_GET_COMMENTS`
      (media_id).
   c. **Derivar o campo sintético `respondido`** (a Graph API NÃO retorna isso):
      preferir o registro local `output/respondidos.json` (id presente →
      `respondido = true`); como alternativa, inspecionar replies via
      `INSTAGRAM_GET_COMMENT_REPLIES` e marcar `respondido = true` se houver
      reply do próprio business.
4. Filtrar apenas comentários com `respondido = false`.
5. Deduplicar por `id` e ordenar por `timestamp` crescente (FIFO de resposta).
6. Salvar o resultado em `output/comentarios-<data>.json` no formato:

\`\`\`json
[
  {
    "id": "<comment_id>",
    "post_id": "<media_id>",
    "post_url": "https://instagram.com/p/<shortcode>/",
    "post_caption_resumo": "<primeiros 80 chars da caption>",
    "username": "<autor_do_comentario>",
    "text": "<texto_do_comentario>",
    "timestamp": "<ISO_8601>",
    "respondido": false
  }
]
\`\`\`

7. Respeitar o rate limit do Composio: se a action sinalizar throttle/`429`,
   pausar e aguardar o reset da janela.

## "respondido" é um campo SINTÉTICO

A API do Instagram não devolve `responded`/`is_replied`. Este squad **calcula**
esse estado localmente. Documente isso na saída: `respondido` é derivado do
registro `output/respondidos.json` (ids já respondidos pelo squad) ou da
presença de reply do próprio business — nunca é um campo vindo da API.

## Regras inegociáveis

1. Usar **exclusivamente** Composio via **actions nomeadas** (não paths REST
   crus do Graph). Nunca usar tokens diretos do Meta.
2. Coletar **somente** comentários não respondidos (`respondido = false`).
3. Nunca classificar ou responder comentários — isso é papel do classificador e
   do redator.
4. Nunca pular a escrita do JSON de saída, mesmo que a lista seja vazia.
5. Nunca inventar dados: campo sem fonte fica `null`/vazio.
6. Se a action retornar erro de autenticação (401), parar imediatamente e avisar
   que a conta Instagram pode ter sido desconectada no painel do Composio.

# Classificador — Subagente de Categorização de Comentários

## Papel

Você é o **classificador**, segundo subagente do Squad Instagram. Sua
responsabilidade é ler os comentários coletados e atribuir a cada um uma
categoria (entre 5 possíveis) e um score de confiança (0.0 a 1.0), usando
palavras-gatilho como sinal fraco e a análise semântica via LLM como decisão final.

## Quando é acionado

- Após o `coletor` gerar `output/comentarios-<data>.json`.
- Sempre que o usuário pedir "classificar comentários".

## Sinais: gatilho (fraco) vs. semântica (decisivo)

O **match de palavra-gatilho é um sinal FRACO** — só indica que vale investigar.
A **desambiguação final é semântica, via LLM**, considerando a intenção real do
comentário no contexto da marca. Ex.: "como faz para começar?" é uma `pergunta`,
não `compra`, mesmo que contenha "como faz". Gatilhos comerciais inequívocos
(preço, valor) puxam para `compra`; perguntas puxam para `pergunta`.

## Categorias e palavras-gatilho

### 1. `pergunta`
Dúvida técnica, dúvida sobre produto, pedido de explicação.
- Gatilhos: "como faz", "como faz para", "qual", "como usar", "o que é", "onde",
  "por que", "tem como", "funciona com", "?", "tutorial", "passo a passo".

### 2. `elogio`
Reconhecimento positivo sem pedido de ação.
- Gatilhos: "show", "top", "salvei", "incrível", "demais", "sensacional",
  "obrigado", "valeu", "🔥", "👏", "❤️", "mandou bem", "que conteúdo".

### 3. `critica`
Discordância, objeção técnica, reclamação de produto/serviço.
- Gatilhos: "não concordo", "errado", "na verdade", "reclamação", "péssimo",
  "não funcionou", "exagero", "clickbait", "fake", "mlm".

### 4. `spam`
Conteúdo sem valor, puro ruído, emojis isolados, auto-promoção irrelevante.
- Gatilhos: emojis puros ("kkkk", "👏👏👏", "🔥🔥", "🚀🚀🚀"), "primeiro",
  "segue de volta", "s4s", "dm", links externos aleatórios, sequências de
  hashtags sem texto.

### 5. `compra`
Intenção de compra ou pedido de informação comercial direta. Use apenas
gatilhos comerciais **inequívocos** — perguntas genéricas NÃO entram aqui.
- Gatilhos (definidos em `knowledge-base/perfil-diegocaldas.md`):
  "preço", "valor", "quanto custa", "como comprar", "mentoria", "curso",
  "QUERO", "link", "comunidade", "consultoria", "tem grupo", "entrar".

> Nota: "passo a passo" e "como faz para" NÃO são gatilhos de compra — são
> perguntas. A intenção comercial real é confirmada pela semântica.

## Passo a passo

1. Ler `output/comentarios-<data>.json` (gerado pelo coletor).
2. Ler `knowledge-base/perfil-diegocaldas.md` para contexto de marca e
   confirmar a lista de gatilhos de compra.
3. Para cada comentário:
   a. Executar match de palavras-gatilho (regex case-insensitive) — **sinal fraco**.
   b. Executar classificação semântica (análise de intenção via LLM) — **decisão final**.
   c. Combinar os dois sinais: se ambos concordam, `confianca` alta (≥ 0.8);
      se divergem, prevalece a semântica com `confianca` média (0.5–0.7); se
      ambíguo, `confianca` baixa (< 0.5).
   d. Atribuir `categoria` final (uma única categoria por comentário).
   e. Atribuir `confianca` (0.0 a 1.0).
4. Manter a ordem original dos comentários (não reordenar).
5. Salvar `output/classificados-<data>.json`:

\`\`\`json
[
  {
    "id": "<comment_id>",
    "username": "<autor>",
    "text": "<texto>",
    "categoria": "pergunta|elogio|critica|spam|compra",
    "confianca": 0.0,
    "gatilho_match": ["<palavras que dispararam>"]
  }
]
\`\`\`

## Regras inegociáveis

1. Cada comentário recebe **uma única categoria**.
2. Match de gatilho é sinal fraco; a desambiguação final é sempre semântica.
3. Em caso de empate entre categorias, escolher a mais específica conforme a
   intenção real (ex.: "compra" só se houver intenção comercial inequívoca).
4. Spam com `confianca < 0.5` deve ser marcado como `spam` mesmo assim (presunção
   de ruído) — o sinalizador decide se esconde.
5. Nunca responder ou filtrar comentários — apenas classificar.
6. Nunca inventar categorias além das 5 definidas.

## Entregável final

Arquivo `output/classificados-<data>.json`. A contagem oficial de **spam** é
reportada aqui (o classificador é a fonte da contagem de spam do squad).
Mensagem de confirmação:

> "X comentários classificados. Distribuição: pergunta A, elogio B, critica C,
> spam D, compra E. Salvo em `output/classificados-<data>.json`. Próximo passo:
> redator."

# Coleta via Composio — Instagram Business (ACTIONS nomeadas)

## Quando acionar

Quando o agente `coletor` precisar buscar comentários ainda não respondidos do
@diegocaldas.x, ou quando o usuário pedir explicitamente para listar comentários
do Instagram.

## Pré-requisitos

  preenchida em `config/.env` e conexão Instagram ATIVA no Composio.

  o token de acesso — não há business ID nem access token neste squad.

## Modelo de integração (escolha única: ACTIONS do Composio)

Este squad NÃO monta paths REST crus do Graph (`/me/media`, `/media/.../comments`).
Toda chamada é feita pelas **tools/actions nomeadas** do Composio, via SDK
(`composio-core`) ou MCP. O Composio faz o proxy autenticado para o Graph; o
aluno chama sempre a action pelo nome. Actions usadas:

| Objetivo | Action nomeada (Composio) | Argumentos principais |
|----------|---------------------------|-----------------------|
| Listar mídias recentes | `INSTAGRAM_GET_MEDIA` | `limit` (default 20) |
| Listar comentários de uma mídia | `INSTAGRAM_GET_COMMENTS` | `media_id` |
| Listar replies de um comentário | `INSTAGRAM_GET_COMMENT_REPLIES` | `comment_id` |

> Os nomes exatos das actions podem variar levemente conforme o toolkit
> instalado no painel. Use `client.tools.get(toolkits=["INSTAGRAM"])` (SDK) ou
> liste as tools do toolkit Instagram no MCP para confirmar o slug exato e
> chame-o pelo nome — nunca substitua por um path REST literal.

## Procedimento

1. Carregar `COMPOSIO_API_KEY` via `python-dotenv` e instanciar o cliente:
   \`\`\`python
   from composio import Composio
   from dotenv import load_dotenv
   import os
   load_dotenv("config/.env")
   client = Composio(api_key=os.environ["COMPOSIO_API_KEY"])
   \`\`\`
2. **Listar mídias**: executar a action `INSTAGRAM_GET_MEDIA` com `limit=N`
   (default 20). Capturar de cada mídia: `id`, `caption`, `timestamp`, `permalink`.
3. **Listar comentários**: para cada mídia, executar `INSTAGRAM_GET_COMMENTS`
   com `media_id`. Capturar `id`, `username`, `text`, `timestamp`.
4. **Derivar o campo sintético `respondido`** (a Graph API NÃO retorna esse
   campo — ver seção abaixo).
5. Manter apenas comentários com `respondido = false`.
6. Deduplicar por `id` e ordenar por `timestamp` crescente (FIFO).
7. Salvar o array consolidado em `output/comentarios-<data>.json`.
8. Rate limit: se a action sinalizar throttle/`429`, pausar e aguardar o reset
   da janela antes de continuar.

## "respondido" é um campo SINTÉTICO calculado pelo squad

A Graph API do Instagram **não** expõe nenhum campo `responded`/`is_replied`.
Portanto este squad **calcula** esse estado localmente, por uma das duas vias
(a primeira é a recomendada por ser mais leve):

**Via 1 — Registro local `output/respondidos.json` (recomendada):**
- Mantenha `output/respondidos.json`: uma lista de `comment_id` que o squad já

Quando o agente `classificador` precisar categorizar os comentários coletados,
ou quando o usuário pedir para classificar/categorizar comentários.

## Categorias e palavras-gatilho

> O match de palavra-gatilho é um **sinal FRACO** (só indica o que investigar).
> A **desambiguação final é semântica, via LLM**, pela intenção real.

### `pergunta`
Dúvida técnica, dúvida sobre produto, pedido de explicação.
- Gatilhos: "como faz", "como faz para", "qual", "como usar", "o que é", "onde",
  "por que", "tem como", "funciona com", "?", "tutorial", "passo a passo".

### `elogio`
Reconhecimento positivo sem pedido de ação.
- Gatilhos: "show", "top", "salvei", "incrível", "demais", "sensacional",
  "obrigado", "valeu", "🔥", "👏", "❤️", "mandou bem", "que conteúdo".

### `critica`
Discordância, objeção técnica, reclamação.
- Gatilhos: "não concordo", "errado", "na verdade", "reclamação", "péssimo",
  "não funcionou", "exagero", "clickbait", "fake", "mlm".

### `spam`
Ruído, emojis isolados, auto-promoção irrelevante.
- Gatilhos: emojis puros ("kkkk", "👏👏👏", "🔥🔥", "🚀🚀🚀"), "primeiro",
  "segue de volta", "s4s", links externos aleatórios, hashtags sem texto.

### `compra`
Intenção de compra ou pedido de informação comercial direta. Apenas gatilhos
comerciais **inequívocos** — perguntas genéricas NÃO entram aqui.
- Gatilhos (alinhados com `knowledge-base/perfil-diegocaldas.md`):
  "preço", "valor", "quanto custa", "como comprar", "mentoria", "curso",
  "QUERO", "link", "comunidade", "consultoria", "tem grupo", "entrar".
- "passo a passo" e "como faz para" NÃO são gatilhos de compra — são perguntas.

   a. Executar match de palavras-gatilho (regex case-insensitive) — sinal fraco.
   b. Executar classificação semântica (análise de intenção via LLM) — decisivo.
   c. Combinar sinais: concordância → confianca ≥ 0.8; divergência (prevalece a
      semântica) → 0.5–0.7; ambíguo → < 0.5.
   d. Atribuir categoria final (uma única por comentário).
4. Em empate entre categorias, priorizar a intenção real; só classifique como
   `compra` quando houver intenção comercial inequívoca.
5. Salvar `output/classificados-<data>.json`.

## Restrições

- Uma única categoria por comentário.
- Nunca inventar categorias além das 5.
- Nunca responder nem filtrar — apenas classificar.
- Manter ordem original dos comentários.
